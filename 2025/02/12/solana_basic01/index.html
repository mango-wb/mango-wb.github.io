Sol的8大核心概念

  • 历史证明（Proof of History, POH）
    • 技术原理:
      • 全局可验证时钟: 将时间编码为可验证的数据结构(sha256)
      • 事件排序: 每个交易自带时间戳哈希，共识底层无需重复验证时序
    • 解决的问题:
      • 传统区块链的节点时钟同步难题(如比特币的10分钟区块间隔)
      • 降低共识机制的时间协调开销
  • Tower BFT共识
    • 技术原理:
      • PBFT优化版: 利用PoH作为可信时间源，减少通信轮次
      • 锁定检查点: 节点对特定的PoH高度达成共识后不可以回滚
  • Gulf Stream协议
    • 技术原理:
      • 交易预推: 验证节点提前交易缓存至下游领导者
      • 零确认交易: 用户可立即发送交易至未来的8个领导节点
    • 性能提升:
      • 减少确认延迟: 交易在到达领导者前已开始处理
      • 网络负载均衡: 避免交易洪峰中冲击单个节点
  • Turbine分块广播
    • 技术原理:
      • UDP+分片传输: 将区块分解为多个小包进行传输
      • 树状拓扑传播: 每个节点负责向特定子节点组转发数据
  • Sealevel并行智能合约环境
    • 技术原理:
      • 交易并行处理: 运行时根据状态依赖关系调度交易
      • 显示状态声明: 合约需预先声明访问的账户，避免资源竞争
  • Cloudbreak水平扩展存储
    • 技术原理:
      • 内存映射文件: 将账户状态存储在随机访问内存中
      • 无锁写入: 通过内存地址偏移实现并行写入
  • Archiver网络
    • 技术原理:
      • 去中心化存储: 历史数据分片存储于轻节点网络
      • 激励模型: 存储提供者通过质押获得网络奖励
  • 本地化费用市场(Local Fee Markets)
    • 技术原理:
      • 资源隔离: 每个状态账户独立设置Gas费用
      • 拥堵控制: 高频访问自动提升交易费用门槛

Solana基础

账户

  • 数据账户
    • 系统所有账户
    • 程序派生账户(PDA)
  • 程序账户，用来存储可执行程序
  • 原生账户，指Solana上的原生程序，例如”System”, “Stake”, 以及”Vote”。

账户结构体

  • Account
    • lamports:u64; : 账户余额(solana余额)
    • data:Vec; : 存储的数据
    • owner: Pubkey; : 存储所有者公钥
    • executable: bool; : 账户是否可执行，如果是可执行，data存储的就是二进制程序；如果不可执行，那data存储的就是数据。
    • rent_epoch: Epoch; : 租金
  • AccountInfo
    • key: &’a Pubkey; 账户公钥，账户标识
    • lamports: Rc<RefCell<&’a mut u64>>; 账户余额(solana余额)
    • data: Rc<RefCell<&’a mut [u8]>>;
    • owner: &’a Pubkey;
    • rent_epoch: Epoch;
    • is_signer: bool; 是否被签名
    • is_writtable:bool; 数据是否可以被修改
    • executable:bool;
  • Account&AccountInfo对比
    • AccountInfo:
      • 更轻量级，包含对区块链上现有账户数据的引用。
      • 用于在Solana程序(智能合约)内部处理账户。
      • 适合在链上处理和操作账户数据
    • Account
      • 更完整的账户表示，包含账户的所有数据副本。
      • 常用于客户端或测试环境中，用于模拟或获取完整的账户状态。
      • 适合离线处理或全局管理账户数据。
  • 要点
    • 账户是用来存放数据的
    • 每个账户都有一个独一无二的地址
    • 每个账户大小不能超过10MB
    • 账户大小是静态的
    • 账户数据存储需要付租金
    • 默认的账户所有者是”系统程序”

solana token解析

solana上的代币

  • 代币是代表对各种资产所有权的数字资产。代币化使得财产权的数字化成为可能，是管理可替代和不可替代资产的基本组成部分。

    • 可代替代币代表同类型和同价值的客户换和可分割资产(USDC)
    • 不可代替代币(NFT)代表不可分割资产的所有权(例如艺术品)

  • SPL(Solana Progranm Library)

    • Token Program(代币程序): 它是SPL中的一个核心程序，用于创建和管理代币。
      • 功能:
        • 创建代币(Mint)
        • 转移代币
        • 冻结或解冻代币
        • 销毁代币
      • 关键指令:
        • InitializeMint: 创建新的代币
        • InitializeAccount: 创建代币账户
        • Transfer: 转移代币
        • MintTo: 铸造代币
        • Burn: 销毁代币
    • Mint Account(铸币账户): 它是代币的铸造账户，用于定义代币的属性和总量
      • 作用:
        • 定义代币的元数据(如名称、符号、小数点位数)
        • 控制代币的总供应量
        • 管理代币的铸造权限
      • 关键字段:
        • supply: 当前代币的总供应量
        • decimals: 代币的小数点位数
        • mint_authority: 代币的铸造权限(可以铸造新代币的地址)
        • freeze_authority: 代币的冻结权限(可以冻结代币账户的地址)
    • Token Account(代币账户): 它是代币的持有账户，用于存储某个用户持有的特定代币
      • 作用:
        • 存储用户持有的代币数量
        • 记录代币的转账和余额变化
      • 关键字段:
        • mint: 关联的Mint Acount地址
        • owner: 代币所有者地址
        • amount: 当前代币的余额
        • delegate: 代币的委托地址(允许其他地址操作代币)
        • state: 账户状态(如是否被冻结)
    • Associated Token Account(关联代币账户): 它是一种特殊的Token Account，它于用户的公钥关联，用于简化代币管理
      • 特点:
        • 每个用户和Mint Account的组合对应一个唯一的Associated Token Account
        • 通过Associated Token Account Program自动创建和管理
      • 作用:
        • 简化代币账户的创建和查找
        • 确保每个用户对每种代币只有一个账户
      • 创建方式:
        • 使用 spl-token create-account 命令或调用 Associated Token Account Program创建
      • 地址计算:
        • Associated Token Account的地址是通过用户公钥和Mint Account地址计算得出的，确保唯一性
